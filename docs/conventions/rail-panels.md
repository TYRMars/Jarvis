# Rail panel conventions

This is the **system-internal interaction standard** for the
right-rail panels in `apps/jarvis-web` (Diff / Tasks / Plan /
Change Report / Files / Terminal / Preview / TODO board / …).
Every new panel should mirror these rules so the workspace rail
reads as one coherent surface, not as a stack of bespoke widgets.

Two reference implementations:

- **`PlanList.tsx`** — read-only checklist driven by `plan_update`
  WS frames. Minimal: status icon + title + status label.
- **`TodosRail.tsx`** — interactive backlog. Adds a composer form
  (input + button), inline priority badge, click-to-cycle status,
  delete affordance, REST-driven hydration with a 503 empty state.

`TodosRail` is the richer reference; copy its structure when a new
panel needs more than a list.

---

## 1. Token vocabulary

Use only these CSS custom properties — defined in `styles.css` at
the top, theme-aware via `:root.dark` overrides.

| token | role |
|---|---|
| `--bg`, `--panel`, `--panel-raised`, `--panel-hover`, `--panel-active` | surface tints, light → heavy |
| `--text`, `--text-muted`, `--text-soft`, `--placeholder` | foreground tiers |
| `--border`, `--border-soft` | hairlines |
| `--accent` (`#d87948`) | primary accent (warm orange) — **default** highlight |
| `--accent-blue` | informational / TODO panel accent |
| `--accent-green` | success / completed |
| `--accent-red` | warning / blocked / destructive |
| `--purple` | change-report accent |
| `--input-bg`, `--input-border`, `--input-focus`, `--placeholder` | form controls |
| `--fs-12`, `--fs-13`, `--fs-14` | the only allowed body type sizes |

**Never** hard-code colour values in component or stylesheet code.
**Never** introduce new size tokens — pick from `--fs-12/13/14`.

Per-panel accent: declare once with `--rail-accent`, e.g.

```css
.rail-card.rail-todos { --rail-accent: var(--accent-blue); }
```

This makes "row hover", "active item border", etc. mappable to one
panel-scoped variable rather than spraying colour literals.

## 2. Panel structure

Every panel is mounted into the rail via the `PanelConfig` array in
`AppWorkspaceRail.tsx`:

```tsx
{
  key: "todos",                      // matches WorkspacePanelKey
  className: "rail-todos",           // declares --rail-accent
  title: tx("panelTodos", "TODOs"),  // i18n with EN fallback
  subtitle: (
    <>
      <TodosCountSpan /> <span>{tx("todosCountLabel", "items in backlog")}</span>
    </>
  ),
  content: <TodosList />,
}
```

Component file exports **two** things:

- `XCountSpan` — drops a `<span id="…-count">{N}</span>` into the
  header subtitle. Subscribes to the store slice.
- `XList` (or `X` for read-only) — the panel body. **No header
  chrome inside this component** — that's the rail's job.

DOM hierarchy generated by the rail:

```
<section class="rail-card rail-todos">
  <header class="rail-card-header">
    <h2>TODOs</h2>
    <p>{count} items in backlog</p>
    <button aria-label="Close TODOs">×</button>
  </header>
  <div class="rail-card-body">
    <!-- your <XList /> renders here -->
  </div>
</section>
```

`.rail-card-body` is the **only** scroll container. Don't add
`overflow: auto` inside the body — internal scroll bars look bad
in the stacked rail.

## 3. Row layout (the grid)

Every itemised panel uses a 4-cell grid:

```css
.todo-item /* and .plan-item */ {
  display: grid;
  grid-template-columns: auto 1fr auto auto;  /* icon | body | label | action */
  align-items: start;
  gap: 8px;
  min-height: 42px;
  padding: 8px 9px;
  border-radius: 8px;
  border: 1px solid transparent;
  background: var(--panel-raised);
  font-size: var(--fs-13);
  line-height: 1.35;
}
```

Column purposes:

1. **icon / status button** — `auto` width, 18–22px tall. Click
   target if the row supports state changes.
2. **body** — `1fr`, holds title + optional note. Always
   `min-width: 0` to allow text wrapping.
3. **label** — `auto`, the small lowercase status pill.
4. **action** — `auto`, optional. Delete / overflow menu.

Read-only panels (`PlanList`) use a 3-cell grid (`auto 1fr auto`)
omitting the action column; everything else stays the same.

Title + note inside the body:

```tsx
<div className="todo-item-body">
  <div className="todo-item-title">
    <span>{item.title}</span>
    {priority && <Badge … />}
  </div>
  {item.notes && <div className="todo-item-note">{item.notes}</div>}
</div>
```

## 4. Status colour vocabulary

The five status names are **reserved** across the rail; pick from
this list when a new panel introduces state:

| status | icon role | tint applied |
|---|---|---|
| `pending` | outline circle, `--text-soft @ 0.6–0.7 opacity` | none (neutral background) |
| `in_progress` | spinning half-circle (`@keyframes plan-spin`), `--accent-blue` | row border + 7% blue background tint |
| `completed` | check, `--accent-green` | title fades to `--text-soft` |
| `blocked` | exclamation, `--accent-red` | row border tint |
| `cancelled` | minus, `--text-soft @ 0.5 opacity` | title strikethrough |

Implement via a single class on the row, e.g.
`<li class="todo-item todo-item-${status}">`. Per-status overrides
target `.<panel>-item-${status} .<element>`.

## 5. Composer form (input + submit)

When a panel needs in-rail content creation, use this exact
structure:

```tsx
<form className="todos-add-form" onSubmit={…}>
  <input
    type="text"
    placeholder={tx("…Placeholder", "…")}
    value={draft}
    onChange={…}
    aria-label={tx("…InputAria", "…")}
  />
  <button
    type="submit"
    disabled={busy || draft.trim().length === 0}
    className="ghost-button"
  >
    {tx("…Add", "Add")}
  </button>
</form>
```

CSS (already shipped — reuse the same class names):

```css
.todos-add-form {
  display: grid;
  grid-template-columns: 1fr auto;
  gap: 6px;
}
```

`.ghost-button` is the labeled-pill variant of `.ghost-icon`. Use
it for any "submit a small thing" action; don't introduce
`.primary-button` / `.btn-primary` etc. for in-rail forms.

## 6. Empty / unavailable state

Two shapes, both built on `.rail-empty`:

**Empty (the panel works, no data yet)**:

```tsx
<div className="rail-empty">
  <span>{tx("…Empty", "Nothing here yet. Do X to populate.")}</span>
</div>
```

**Unavailable (the backend feature is off / 503)**:

```tsx
<div className="rail-empty">
  <strong>{tx("…Unavailable", "Feature unavailable")}</strong>
  <span>{tx("…UnavailableHint", "Set X to enable.")}</span>
</div>
```

`.rail-empty` is centred, padded, soft-grey. Don't override font
size or padding — if the message is too long, shorten it.

## 7. i18n discipline

Everything user-facing must walk through `t(...)`. Use the local
`tx(key, fallback)` helper any time the key may be missing in
the active locale — it falls through to the English fallback
instead of leaking the raw key.

Naming pattern: `<feature><Affordance>` and `<feature><Affordance>Aria`
for screen-reader-only strings.

Examples (already in `utils/i18n.ts`):

| key | EN | ZH |
|---|---|---|
| `todosAdd` | "Add" | "添加" |
| `todosAddPlaceholder` | "Add a TODO and press enter…" | "输入待办，按回车添加…" |
| `todosEmpty` | "No TODOs yet…" | "暂无待办…" |
| `todosUnavailable` | "TODOs unavailable" | "持久化待办未启用" |
| `todoStatusPending` | "todo" | "待办" |

**Confirmation prompts** with substitutions use `{title}`-style
placeholders + `String.replace`:

```ts
const tmpl = tx("todosDeleteConfirm", "Delete TODO {title}?");
const msg = tmpl.replace("{title}", item.title);
if (!confirm(msg)) return;
```

## 8. Icons

- 14×14 `<svg viewBox="0 0 24 24" stroke="currentColor" stroke-width="1.8">`
  inside row buttons.
- 13×13 inside menu items (panel-menu icons).
- 28×28 hit area for `.ghost-icon`; 22×22 for inline action icons.
- Always `aria-hidden="true"` on the SVG; `aria-label` on the
  parent button.
- Stroke-only, no fills. Use `currentColor`; let CSS drive colour.

---

## 9. Inline-style ban

`style={{...}}` on JSX is reserved for **dynamic values that can't
be classes** — typically computed pixel offsets driven by measured
DOM. Do not use inline `style` for:

- colours (use a class + CSS var)
- spacing (use a class)
- font sizes (use `--fs-*` via a class)
- borders (use a class)

If a class doesn't exist for what you need, **add one to
`styles.css`** rather than inlining.

---

## 10. Live updates

The chat WebSocket is the canonical event bus. New panels that
need real-time data subscribe via:

1. REST hydrate on mount (`useEffect`).
2. WS frame handler in `services/frames.ts` dispatches into store
   actions (`upsertX`, `removeX`, `setX`).
3. Optimistic updates immediately after a successful REST mutation
   so users see their own action reflected without waiting for the
   broadcast round-trip.

See `services/frames.ts` `case "todo_upserted":` / `case
"todo_deleted":` for the pattern. Never call `useEffect` polling
loops to drive panel state.

---

## 11. Accessibility floor

- Row click targets ≥ 22×22 px.
- All buttons have `aria-label`.
- Lists use `role="list"` (the default for `<ol>` / `<ul>`); items
  use the implicit `listitem` role.
- Form inputs always have either a visible `<label>` or
  `aria-label`.
- Status badges are decorative — they duplicate info already in
  the icon class — so they don't need `aria-live`.

## 12. Checklist before merging a new panel

- [ ] Two named exports: `XCountSpan` + `X(List)`.
- [ ] Mounted in `AppWorkspaceRail.tsx` `panels` array with a
      `WorkspacePanelKey` and `--rail-accent`.
- [ ] CSS lives in `styles.css` under a `=== Panel name ===`
      banner comment, between the existing panel sections (not
      sprinkled across the file).
- [ ] All visible text walks `t(...)` with EN + ZH entries.
- [ ] Empty / unavailable states render via `.rail-empty`.
- [ ] No `style={{...}}` for static visual properties.
- [ ] Live updates wire through `frames.ts` if applicable.
- [ ] Vitest covers the panel's frame handlers (see
      `frames.test.ts`).
- [ ] This document mentions the new panel in §0 if it joins as
      a reference impl.
